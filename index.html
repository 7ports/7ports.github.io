Project Voltron
MCP server · Claude Code agent teams
MCP Server

Scaffold any project with battle-tested subagent definitions for Claude Code. Voltron provides a coordinated team of specialists — a scrum-master that decomposes work, a project-planner that designs architecture, and domain experts that implement it. Specialist agents run in Docker containers for blast-radius isolation, and the whole system self-improves through post-session reflections.

  • 11 specialist agents spanning Unity, web/fullstack, and general software development
  • Self-improving: post-session reflections feed a GitHub Actions pipeline (Mon/Wed/Fri) that patches templates and opens PRs for review
  • Auto-update hook silently upgrades installed agent files at the start of every session
  • Real-time progress dashboard with Chrome MCP integration — opens and focuses automatically when work begins
  • Integrates with Project Alexandria — agents consult the shared knowledge base before any tool installation (mandatory)

Project Alexandria
Shared knowledge base · all Claude instances
Knowledge Base

A collaboratively maintained library of tooling setup guides shared across Claude instances via MCP. Every Claude instance with Alexandria installed becomes both a consumer and a contributor — when one instance sets up a tool and documents the process, every other instance benefits. Setup knowledge accumulates across projects and sessions instead of being rediscovered from scratch.

  • Agents call quick_setup before any tool install — mandatory, not optional; update_guide after completion
  • 13+ guides: Git, GitHub, Firebase, AWS CLI, Claude in Chrome, Claude Preview, Beads, Memory MCP, CoPlay, and more
  • Strict content boundary — only non-project-specific, reusable knowledge belongs here; project-specific decisions stay in CLAUDE.md
  • No database, no external dependencies — plain markdown files readable by humans and Claude alike
  • Companion to Voltron — all Voltron agents reference Alexandria; scrum-master calls get_project_setup_recommendations on every new project

Project Pepper
2D card game · Unity 6 · WebGL
Unity Game

A strategic 2D card game built in Unity 6 for WebGL. Players compete on a 4×4 grid using a hand of cards — each card has unique effects that alter the board state. Three AI difficulty levels challenge players as they try to claim four in a row on the expanded grid. Firebase Analytics tracks all major gameplay events via a custom JavaScript bridge.

  • Hand-of-cards mechanic on a 4×4 Tic-Tac-Toe grid — cards as ScriptableObjects, each with distinct board effects
  • Three AI opponent difficulty levels (Easy / Medium / Hard) with distinct decision-making strategies
  • Firebase Analytics via custom JS bridge: FirebaseWebGL.cs → FirebaseAnalytics.jslib → browser Firebase SDK — fully WebGL-compatible, guarded with #if UNITY_WEBGL && !UNITY_EDITOR
  • Universal Render Pipeline (URP) with custom sprite materials; TextMesh Pro for all UI text
  • Full Voltron agent team: scrum-master · csharp-dev · scene-architect · shader-artist · build-validator · asset-manager

Project Hammer
Real-time ferry tracker · Toronto Harbour
Web App

Real-time map showing Toronto Island Ferry positions using live AIS vessel data. A Node.js backend on Fly.io proxies the aisstream.io WebSocket (which blocks direct browser connections) and relays position updates as SSE. The React PWA renders ferries gliding across the harbour with 60fps smooth interpolation via lerp() + requestAnimationFrame.

  • Live AIS position tracking for all four Toronto Island Ferry vessels (Sam McBride, Wm Inglis, Thomas Rennie, Marilyn Bell I) filtered by MMSI
  • SSE over WebSocket relay — EventSource auto-reconnects, no client-side WebSocket library needed; unidirectional AIS data is a perfect fit
  • 60fps smooth vessel movement interpolating between ~10s AIS ping intervals using lerp() + requestAnimationFrame
  • MapLibre GL JS v5 + react-map-gl v8 + MapTiler Ocean chart style — fully open source, no proprietary map SDK lock-in
  • Full production infrastructure: Terraform (S3 + CloudFront + ACM + Route53) · Fly.io yyz region · GitHub Actions CI/CD · PWA with service worker

How it fits together

The Ecosystem

Voltron + Alexandria

The two MCP servers are designed to work together. Voltron provides the agent team; Alexandria provides the knowledge base. Every Voltron agent is required to call quick_setup before installing any tool. After setup, they call update_guide. Tool knowledge flows from sessions back into Alexandria for every future project.

Voltron + Projects

Project Pepper (Unity) and Project Hammer (web) are both developed using the full Voltron agent workflow — scrum-master for planning, domain specialists for implementation, all coordinated via the progress dashboard. They serve as real-world validation for the agent templates.

Self-Improving Loop

Specialist agents submit post-session reflections at phase completion and session end. The scrum-master submits automatically. A GitHub Actions workflow processes reflections three times a week, applies targeted improvements to agent templates, and opens PRs for review — closing the loop between usage and improvement.
