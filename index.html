Voltron is an MCP server that gives Claude Code a coordinated team of specialist subagents. A scrum-master decomposes work, a project-planner designs architecture, and domain experts implement — each running in its own Docker container. Agents self-improve through a feedback loop: post-session reflections accumulate and a GitHub Actions workflow applies targeted improvements to the templates three times a week, opening PRs for review before anything reaches main.

  • 11 specialist agents across Unity (scene-architect, csharp-dev, shader-artist, build-validator, asset-manager) and web/fullstack (fullstack-dev, devops-engineer, ui-designer, qa-tester)
  • Self-improving reflection loop — scrum-master submits post-session feedback automatically; GitHub Actions (Mon/Wed/Fri 10:00 UTC) patches templates and opens a PR
  • Auto-update hook silently upgrades installed agent files, Dockerfile, and run script at the start of every session
  • Real-time progress dashboard with Chrome MCP integration — tab opens and refocuses automatically when work begins
  • Integrates with Project Alexandria — all agents call quick_setup before any tool installation; mandatory, not optional

Alexandria is a collaboratively maintained library of tooling setup guides shared across Claude instances via MCP. Every Claude instance that has Alexandria installed becomes both a consumer and a contributor. When one instance sets up a tool and documents the process, every other instance benefits — error/fix pairs, platform quirks, version notes, and working commands accumulate across sessions instead of being rediscovered from scratch every time.

  • Agents call quick_setup before any tool installation — mandatory, not optional; they call update_guide after completing setup to document what they learned
  • 13+ guides covering Git, GitHub, Firebase, AWS CLI, Claude in Chrome, Claude Preview, Beads, Memory MCP, CoPlay, and more
  • Strict content boundary — only non-project-specific, reusable knowledge; project-specific decisions stay in CLAUDE.md
  • No database, no external dependencies — guides are plain markdown files readable and editable by both humans and Claude
  • Companion to Voltron — all Voltron agents reference Alexandria; scrum-master calls get_project_setup_recommendations when starting any new project

Project Pepper Unity Game
2D strategic card game for WebGL
Alpha
Play on itch.io GitHub

A strategic 2D card game built in Unity 6 for WebGL. Players compete on a 4×4 grid using a hand of cards — each card has unique effects that alter the board state. Three AI difficulty levels (Easy, Medium, Hard) challenge players as they try to claim four in a row. Firebase Analytics tracks all major gameplay events via a custom JavaScript bridge that connects Unity C# to the browser Firebase SDK, fully guarded for WebGL-only execution.

  • Hand-of-cards mechanic layered over a 4×4 Tic-Tac-Toe grid — cards are ScriptableObjects, each with distinct board effects that create strategic depth
  • Three AI difficulty levels (Easy / Medium / Hard) with distinct decision-making strategies and evaluation heuristics
  • Firebase Analytics via custom JS bridge: FirebaseWebGL.cs → FirebaseAnalytics.jslib → browser Firebase JS SDK — correctly guarded with #if UNITY_WEBGL && !UNITY_EDITOR throughout
  • Universal Render Pipeline (URP) v17.2 with custom sprite materials; TextMesh Pro for all UI text; singleton-based controller architecture
  • Fully managed by the Voltron agent team: scrum-master · csharp-dev · scene-architect · shader-artist · build-validator · asset-manager

Real-time map showing Toronto Island Ferry positions using live AIS vessel data from aisstream.io. A Node.js backend on Fly.io (Toronto region) proxies the AIS WebSocket — which blocks direct browser connections — and relays vessel position updates as Server-Sent Events. The React PWA renders ferries gliding across the harbour with smooth 60fps interpolation between the ~10-second AIS ping intervals.

  • Live AIS tracking for all four Toronto Island Ferry vessels by MMSI: Sam McBride · Wm Inglis · Thomas Rennie · Marilyn Bell I
  • SSE relay over WebSocket — EventSource auto-reconnects, no client-side WebSocket library; unidirectional AIS data is a natural fit for SSE
  • 60fps smooth vessel movement with position interpolation between AIS pings using lerp() + requestAnimationFrame
  • MapLibre GL JS v5 + react-map-gl v8 + MapTiler Ocean chart style — fully open source, no proprietary map SDK lock-in
  • Full production infrastructure: Terraform (S3 + CloudFront + ACM + Route53) · Fly.io yyz region · GitHub Actions CI/CD · PWA with service worker

How it fits together

The Ecosystem


Voltron + Alexandria

The two MCP servers are designed to work together. Voltron provides the agent team; Alexandria provides the knowledge base. Every Voltron agent calls quick_setup before installing any tool. After setup, they call update_guide. Tool knowledge flows from sessions into Alexandria for every future project.

Voltron in Production

Project Pepper and Project Hammer are both developed using the full Voltron workflow — scrum-master for planning, specialist agents in Docker for implementation, live progress dashboard for visibility. They serve as real-world validation for the agent templates and generate the reflections that feed the self-improvement loop.

Self-Improving Loop

Specialist agents submit post-session reflections at phase completion and session end. A GitHub Actions workflow processes them three times a week (Mon/Wed/Fri at 10:00 UTC), applies targeted improvements to agent templates, and opens PRs for human review before anything reaches main.
